# Глоссарий терминов проекта «Умное бронирование»

| Термин | Определение | Синонимы | Примечание |
|--------|-------------|----------|------------|
| **Агрегатор столиков** | Сервис, который собирает информацию о свободных столиках в ресторанах и позволяет гостям бронировать их онлайн | Платформа бронирования, booking aggregator | В проекте — основная система |
| **Гость** | Пользователь сервиса, который ищет и бронирует столики | Клиент, посетитель, пользователь | Основной актор системы |
| **Администратор ресторана** | Сотрудник ресторана, управляющий бронями в своем заведении | Менеджер, хостес | Имеет доступ к админ-панели |
| **Владелец ресторана** | Лицо, управляющее настройками ресторана в системе | Управляющий, ресторатор | Настраивает правила депозита |
| **Бронь (бронирование)** | Запись о том, что гость зарезервировал столик на определенное время | Reservation, booking | Содержит дату, время, кол-во гостей |
| **No-show** | Ситуация, когда гость не приходит по забронированному столику без отмены | Неявка | Основная бизнес-проблема проекта |
| **Депозит** | Предоплата, которую гость вносит для гарантии брони | Залог, предоплата | Может быть возвратным или невозвратным |
| **Умный депозит** | Динамический расчет суммы депозита в зависимости от дня недели, времени и популярности ресторана | Адаптивный депозит, dynamic deposit | Ключевая функция модуля |
| **Платёжный шлюз** | Внешняя система для обработки онлайн-платежей | Эквайринг, payment gateway | В проекте — гипотетический сервис |
| **API** | Интерфейс для взаимодействия между разными системами | Application Programming Interface | Используется для связи с платежным шлюзом |
| **JSON** | Формат обмена данными между клиентом и сервером | JavaScript Object Notation | Используется в API-запросах |
| **Стейкхолдер** | Заинтересованное лицо, влияющее на требования к системе | Stakeholder | Гости, администраторы, владельцы |
| **Use Case** | Описание того, как пользователь взаимодействует с системой для достижения цели | Вариант использования | Документируется в диаграммах UML |
| **User Story** | Короткое описание функции с точки зрения пользователя | Пользовательская история | Формат: «Как [роль] я хочу [действие] чтобы [ценность]» |
| **BPMN** | Нотация для моделирования бизнес-процессов | Business Process Model and Notation | Используется для сквозных процессов |
| **UML** | Унифицированный язык моделирования | Unified Modeling Language | Используется для диаграмм (Use Case, Activity) |
| **ER-диаграмма** | Модель данных, показывающая сущности и связи между ними | Entity-Relationship diagram | Описывает структуру базы данных |
| **MVP** | Минимально жизнеспособный продукт | Minimum Viable Product | Первая версия с ключевым функционалом |
| **Критерии приемки** | Условия, при которых задача считается выполненной | Acceptance Criteria | Проверяются при тестировании |
| **Идемпотентность** | Свойство операции, при котором многократный вызов дает тот же результат, что и однократный | Idempotency | Важно для платежей — повтор не создаст дубль |
