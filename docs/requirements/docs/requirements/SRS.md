# Спецификация требований (Software Requirements Specification)
## Модуль «Умное бронирование» v1.0

### 1. Введение
#### 1.1 Цель
Разработать модуль динамического расчета депозита для агрегатора бронирования столиков.

#### 1.2 Целевая аудитория
- **Гости** — пользователи, ищущие столики
- **Администраторы ресторанов** — сотрудники, управляющие бронями
- **Владельцы ресторанов** — принимают решения по настройкам

### 2. Общее описание
#### 2.1 Контекст
Модуль интегрируется в существующий сервис поиска и бронирования столиков.

#### 2.2 Функции продукта
- Расчет депозита по динамическим правилам
- Отображение правил расчета для гостя
- Управление настройками депозита для ресторана
- Обработка отмен и возвратов

### 3. Функциональные требования

#### 3.1 Модуль расчета депозита
| ID | Требование | Приоритет |
|----|------------|-----------|
| FR-01 | Система должна рассчитывать депозит на основе дня недели, времени и рейтинга ресторана | Высокий |
| FR-02 | Правила расчета должны настраиваться в админ-панели ресторана | Средний |
| FR-03 | Система должна показывать гостю, из чего сложилась сумма депозита | Высокий |

#### 3.2 Управление бронями
| ID | Требование | Приоритет |
|----|------------|-----------|
| FR-04 | Гость может отменить бронь в личном кабинете | Высокий |
| FR-05 | При отмене за 24+ часа — возврат 100% депозита | Высокий |
| FR-06 | При отмене менее чем за 24 часа — возврат 50% | Средний |

#### 3.3 Административная панель
| ID | Требование | Приоритет |
|----|------------|-----------|
| FR-07 | Администратор видит все брони в едином списке | Высокий |
| FR-08 | Администратор может подтверждать/отклонять брони | Высокий |
| FR-09 | Администратор может настраивать правила депозита | Низкий |

### 4. Нефункциональные требования
| ID | Требование | Описание |
|----|------------|----------|
| NFR-01 | Производительность | Страница бронирования загружается < 3 сек |
| NFR-02 | Доступность | Сервис доступен 99.5% времени |
| NFR-03 | Безопасность | Данные карт не хранятся в системе |
